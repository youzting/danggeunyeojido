# 성능 측정 결과

## 측정 목적

검색 속도를 다음 구간으로 나눠 확인한다.

- `planningTimeMs`: 최적 반경과 검색 거점 경로 계산 시간
- `listingFetchTimeMs`: 검색 공급자에서 결과를 가져오는 시간
- `totalElapsedTimeMs`: 서버 내부 전체 처리 시간
- `httpMs`: 클라이언트에서 관측한 HTTP 왕복 시간

응답 DTO에 위 수치를 포함해 화면과 API에서 바로 확인할 수 있게 했다.

## 개선 1. 거리 행렬 캐싱

기존 구현은 거점 후보를 평가할 때마다 Haversine 거리 계산을 반복했다.

개선 후에는 검색 1회 단위로 `RegionDistanceMatrix`를 생성해 지역 간 거리를 재사용한다.

효과:

- 후보 반경 평가 중 반복 삼각함수 계산 감소
- 거점 선택 로직에서 거리 조회 비용 안정화
- 이후 지역 수 증가 시 캐시/영속화 확장 가능

## Mock 공급자 기준

측정 조건:

- Profile: default
- Provider: `mock`
- Keyword: `맥북`
- API: `GET /api/search/nationwide?keyword=맥북`
- 지역 앵커: 24개
- 최적 반경: 50km
- 검색 거점: 12개
- 결과 수: 24건

측정 결과:

| Run | HTTP ms | Planning ms | Listing ms | Total ms |
| --- | ---: | ---: | ---: | ---: |
| 1 | 449 | 96 | 2 | 101 |
| 2 | 57 | 47 | 0 | 48 |
| 3 | 51 | 43 | 0 | 43 |
| 4 | 44 | 36 | 0 | 37 |
| 5 | 55 | 34 | 0 | 34 |

해석:

- 첫 요청은 JVM/JIT/DispatcherServlet 초기화 영향으로 느리다.
- 워밍업 후 서버 내부 전체 처리 시간은 34~48ms 범위다.
- Mock 공급자에서는 대부분의 시간이 최적 거점 계산에서 발생한다.

## 실제 공개 웹 데이터 공급자 기준

측정 조건:

- Profile: `daangn`
- Provider: `daangn-public-web`
- Keyword: `맥북`
- API: `GET /api/search/nationwide?keyword=맥북&maxStops=3`
- 검색 거점: 3개
- 최적 반경: 200km
- 커버리지: 100%
- 결과 수: 6건

측정 결과:

| HTTP ms | Planning ms | Listing ms | Total ms |
| ---: | ---: | ---: | ---: |
| 660 | 47 | 445 | 495 |

해석:

- 실제 공급자에서는 병목이 거점 계산이 아니라 외부 HTTP 요청이다.
- 3개 거점만 조회해도 `listingFetchTimeMs`가 전체 처리 시간의 대부분을 차지한다.
- 실제 전국 검색에서 거점 수가 증가하면 병렬화, 캐싱, rate limit 제어가 필요하다.

## 실제 공급자 연결 시 발견한 이슈

### 1. 지역명 해석 불안정

`대전 서구`처럼 시군구 이름만으로 지역을 해석하면 중복 지명 때문에 엉뚱한 동네가 선택될 수 있다.

해결:

- 내부 `Region` id 기준으로 대표 동네 id를 고정 매핑했다.
- 현재 주요 smoke test 거점은 다음 대표 동네를 사용한다.

```text
daejeon-seogu -> 둔산동-5793
gyeongbuk-pohang -> 죽도동-3121
jeonnam-mokpo -> 상동-2766
```

### 2. 일부 동네 검색 라우트 500 응답

지역 해석 API의 첫 결과가 검색 라우트에서 500을 반환하는 경우가 있었다.

예:

```text
옥암동-2770
오천읍-3098
```

해결:

- smoke test용 대표 동네를 응답이 안정적인 동네로 교체했다.
- 외부 공급자 실패는 전체 검색 실패로 전파하지 않고 해당 거점 결과만 빈 목록으로 처리한다.

## 다음 개선 목표

1. `Region` 엔티티에 실제 공급자용 대표 동네 id를 저장한다.
2. 외부 검색 요청을 병렬화하되 동시성 제한을 둔다.
3. 검색어 + 거점 id 기준 캐시를 추가한다.
4. 중복 URL 기준 결과 중복 제거를 강화한다.
5. 3개, 6개, 12개 거점 기준 P50/P95 응답 시간을 반복 측정한다.
