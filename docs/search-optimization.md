# 검색 거점 최적화

## 문제 정의

이 프로젝트의 핵심은 검색어 자체가 아니라 **전국을 검색하기 위해 어느 지역을 기준으로 몇 번 검색해야 하는가**다.

지역 기반 검색 서비스에서는 한 번의 검색이 특정 지역과 반경 안에서만 유효하다고 가정한다. 전국 검색을 만들려면 여러 지역을 기준으로 반복 검색해야 한다.

이를 다음 문제로 정의했다.

```text
주어진 지역 앵커 목록과 검색 반경에서,
전국 지역을 최대한 적은 검색 거점으로 커버하는 경로를 찾는다.
```

## 입력값

현재 프로토타입의 입력은 다음과 같다.

- `keyword`: 사용자 검색어
- `regions`: 검색 기준점 후보 지역 목록
- `radiusCandidates`: 자동 평가할 후보 반경 목록
- `maxStops`: 최대 검색 거점 수

현재 후보 반경:

```text
8, 10, 15, 20, 25, 30, 40, 50, 60, 80, 100, 120, 150, 200km
```

기본 최대 검색 거점 수:

```text
24
```

목표 검색 거점 수:

```text
12
```

## 출력값

검색 최적화 결과는 다음 값을 반환한다.

- `optimizedRadiusKm`: 자동 선택된 검색 반경
- `searchRegionCount`: 실제 검색에 사용할 거점 수
- `coveragePercent`: 전체 지역 커버율
- `totalMoveKm`: 거점 순회 기준 예상 이동거리
- `steps`: 거점별 커버리지 변화
- `remainingRegions`: 커버하지 못한 지역

## 알고리즘 흐름

현재 구현은 `NationwideSearchService`에 있다.

1. 후보 반경을 작은 값부터 순회한다.
2. 각 반경마다 검색 거점 경로를 만든다.
3. 시작 지역이 명시되면 해당 지역을 첫 거점으로 사용한다.
4. 시작 지역이 없으면 신규 커버 지역 수가 가장 큰 지역을 첫 거점으로 자동 선택한다.
5. 이후에는 아직 선택하지 않은 지역 중 새로 커버하는 지역 수가 가장 큰 지역을 고른다.
6. 새로 커버하는 지역 수가 같으면 직전 거점과의 이동거리가 짧은 지역을 우선한다.
7. 모든 지역이 커버되면 해당 반경 평가를 종료한다.
8. 전국 커버가 되고 목표 거점 수 이하인 첫 반경을 최적 반경으로 선택한다.
9. 조건을 만족하는 반경이 없으면 가장 좋은 fallback 결과를 사용한다.

## 거점 선택 기준

거점 선택의 1순위는 신규 커버 지역 수다.

```text
newlyCoveredCount(candidate, radius)
```

후보 지역을 기준으로 `radius` 안에 들어오면서 아직 커버되지 않은 지역 수를 계산한다.

예시:

```text
서울 강남구 50km 검색
→ 서울, 경기 일부, 인천 일부를 한 번에 커버
```

2순위는 이동거리다.

```text
distance(previousRegion, candidateRegion)
```

신규 커버 수가 같다면 직전 거점에서 더 가까운 지역을 선택한다. 같은 커버리지라면 이동 비용이 낮은 경로가 더 낫기 때문이다.

## 반경 최적화 기준

검색 반경은 작을수록 지역 검색 의도에 가깝다. 하지만 너무 작으면 검색 거점 수가 급격히 늘어난다.

반대로 검색 반경이 너무 크면 적은 거점으로 전국을 커버할 수 있지만, 지역 기반 검색의 장점을 잃을 수 있다.

따라서 현재 기준은 다음과 같다.

```text
전국 100% 커버가 가능하면서,
검색 거점 수가 목표치 이하인 가장 작은 후보 반경
```

현재 샘플 데이터에서는 다음 결과가 나온다.

```text
optimizedRadiusKm = 50
searchRegionCount = 12
coveragePercent = 100.0
totalMoveKm = 1483.5
```

## 왜 탐욕 알고리즘인가

이 문제는 Set Cover와 유사하다. 모든 지역을 덮는 최소 거점 조합을 찾는 문제로 보면 완전 탐색 비용이 커질 수 있다.

현재 프로젝트는 포트폴리오용 프로토타입이며, 지역 앵커와 외부 검색 요청이 늘어날 가능성을 고려해야 한다. 그래서 다음 이유로 탐욕 알고리즘을 선택했다.

- 구현이 단순하고 설명 가능하다.
- 지역 수가 늘어나도 빠르게 근사해를 구할 수 있다.
- 검색 거점이 많아질수록 외부 요청 비용이 커지므로 신규 커버 수 우선 전략이 실용적이다.
- 실제 운영 전 단계에서 병목과 개선 포인트를 확인하기 쉽다.

## 현재 한계

현재 알고리즘은 최적해를 보장하지 않는다.

한계:

- 전체 이동거리 최소 경로가 아닐 수 있다.
- 지역별 실제 거래량을 고려하지 않는다.
- 인구 밀도와 검색 결과 밀도를 반영하지 않는다.
- 바다, 산악 지역, 생활권 같은 실제 이동/검색 특성을 고려하지 않는다.
- 현재 지역 앵커 수가 작아 실제 전국 읍면동 단위 데이터와 차이가 있다.

## 개선 방향

### 1. 거리 행렬 캐싱

현재는 거리 계산이 필요할 때마다 Haversine 공식을 호출한다.

지역 수가 늘어나면 거리 행렬을 미리 만들 수 있다.

```text
distance[fromRegionId][toRegionId]
```

효과:

- 반복 삼각함수 계산 제거
- 후보 반경 평가 속도 개선
- 거점 선택 로직 단순화

### 2. 점수 기반 거점 선택

현재는 신규 커버 수와 이동거리만 본다.

실제 검색 품질까지 고려하려면 다음 식으로 확장할 수 있다.

```text
score =
  newlyCoveredCount * 10
  + expectedListingDensity
  - moveDistanceKm * 0.03
  - duplicatedCoveragePenalty
```

이렇게 하면 단순 지리 커버뿐 아니라 매물이 많이 나올 가능성이 높은 지역을 우선할 수 있다.

### 3. 권역 기반 2단계 탐색

전국을 바로 읍면동 단위로 탐색하면 후보가 많아질 수 있다.

개선안:

1. 서울/수도권, 충청, 호남, 영남, 강원, 제주 같은 권역 단위로 1차 거점을 선택한다.
2. 커버되지 않은 지역만 2차로 세분화한다.

효과:

- 초기 검색 요청 수 감소
- 넓은 지역 커버 후 미세 조정 가능

### 4. 외부 검색 결과 기반 재탐색

실제 검색 공급자가 연결되면 결과 수에 따라 다음 거점을 조정할 수 있다.

예시:

- 특정 거점에서 결과가 충분히 많으면 인접 거점 검색을 줄인다.
- 결과가 적은 권역은 더 촘촘한 반경으로 재검색한다.
- 중복 결과가 많은 거점은 우선순위를 낮춘다.

## 포트폴리오에서 강조할 점

이 프로젝트의 핵심은 단순히 검색 화면을 만든 것이 아니라 다음 의사결정을 코드와 문서로 남긴 것이다.

- 지역 검색을 전국 검색 문제로 재정의했다.
- 전국 커버를 거점 선택 문제로 모델링했다.
- 최적해 대신 설명 가능한 근사 알고리즘을 선택했다.
- 실제 외부 연동 전에도 교체 가능한 Repository 구조를 만들었다.
- 성능 병목과 개선 방향을 미리 문서화했다.
